## ① 是什么 + 定位 + license

**一句话**：一套**面向 Web UI**（字色 / 背景 / 边框）的开源调色板，把 13 个色相各切成 10 个亮度档，号称「同一亮度档跨色相的感知亮度恒定」。定位是给界面设计当**通用色板**，不是品牌色系统（README:3、:10）。

- **作者/背书**：个人项目（非机构），但生态认可度高——**Mantine、Excalidraw 等知名前端库的默认色板就源自 open-color**（其影响力主要在前端圈，不在 deck 圈）。
- **核心设计目标 4 条**（README:5-10·原文）：① 每个色都有恰当用途 ② 提供 UI 通用色 ③ 色本身好看且彼此和谐 ④ **同亮度档感知亮度恒定**。
- **⚠️ 作者自己的红线**（README:14·原文）：「颜色未来可能变，**不建议把 open-color 当主品牌识别色**（main identity color）」——这条直接决定了它对我们的用法（见 §3/§4）。
- **license 干净**：MIT，可商用 / 可改 / 可二次分发，只需保留版权声明。对「当配色资产打包进 skill」无障碍。

## ② 有什么（色板结构 · 已读 json 核对）

**总量 = 132 色**：`white` + `black`（各 1 单值）+ **13 个彩色色相 × 10 档**（0→9）。彩色色相全部确认为 10 档，无例外。

**13 色相**（按 json 顺序，附 0 档最浅 → 9 档最深的端点）：

**档位语义（约定俗成·非官方明文，来自 README 示例 + 前端社区惯例）**：
- `0~2` = 浅背景 / hover 底；`5~6` = 主用（按钮、图标、数据色主体）；`7~9` = 文字 / 边框 / 描边。
- README「How to Use」示例即印证：背景用 `gray-0`、正文用 `gray-7`、链接用 `teal-7`/`indigo-7`（README:74-87）。
- **命名约定**：`oc-(色相)-(档号)`，跨格式统一（CSS `--oc-blue-6`、Sass `$oc-blue-6`、Less `@oc-blue-6`）。

**关键缺口 = 没有语义层**：open-color **只给「色相×档位」的物理色板，不给 primary/success/danger/warning 这类语义 token**（全仓搜 `semantic`/`accent`/`primary`/`danger` 零命中，README 仅那一句红线）。语义映射要使用方自己定。这点跟麦肯锡规范是镜像关系（见 §3）。

**供给格式 10 种**（根目录 `open-color.*`，全部由 `compile-templates.js` 从模板生成）：`.json`（真相源）/`.css`(CSS 变量) / `.scss` / `.less` / `.styl` / `.tex`(LaTeX) / `.oco`(Open Color Tools) / `.rcpx`(PowerPaint) / `.d.ts`(TS 类型) ＋ `docs/asset/download/` 里的 **`.sketchpalette`（Sketch 设计工具）** 与历史版 `.aco/.clr`。**注意：无 `.thmx`（PowerPoint 主题）、无 `.pptx`** —— 给 deck 用要自己搬色值。

## ③ 怎么用进咨询级 deck（vs 麦肯锡配色规范）

把它和明珠 Mck-skill 的配色规范（出处 `constants.py` + `brand-guide.md`，见 `specs/PPT方法论/02_页型手法精华.md` §三）正面对照——**两者是「物理色库」与「语义规范」的互补，不是替代**：

| 维度 | 麦肯锡规范（咨询级目标态） | open-color（本库） | 对 deck 的含义 |
|---|---|---|---|
| **主色** | NAVY `#051C2C` 唯一主色 | **无主色概念**（作者明令别当 identity） | open-color **顶不了** NAVY 的位；主色得自定 |
| **强调色** | 蓝/绿/橙/红 4 个，且「仅 3+ 并列项才准用」 | 13 色相任选，无使用纪律 | open-color 给**候选池**，纪律得我们加 |
| **灰阶** | 4 级（正文#333/次要#666/分隔#CCC/底#F2F2F2） | **10 级** gray（#f8f9fa→#212529） | open-color 灰阶更细，可当**取色梯子**反挑出咨询级 4 档 |
| **数据色 vs 品牌色** | 强调四色＝并列项数据色，克制 | 130 彩色＝纯数据/装饰色 | **这是 open-color 的真正落点**：当**数据系列配色盘** |
| **废弃色** | CYAN 明令废弃 | cyan 是正式色相之一 | 若采用须**手动剔 cyan**，否则违背明珠规范 |

**用法定调（关键区分：数据色 vs 品牌色）**：
- ✅ **数据色（推荐用法）**：图表里多系列 / 分类配色，从 open-color 的 `5~7` 档挑一组**感知亮度相近**的色（这正是它「同档亮度恒定」设计目标的价值点）——比临时拍脑袋调色稳。waterfall/Mekko 这类多色块图能直接受益。
- ✅ **灰阶取色梯**：用 10 级 gray 当背景/分隔/边框的**取色参考**，再压成咨询级常用的 3~4 档。
- ❌ **品牌色 / 主色**：**别用**。作者自己说了色会变、且咨询级 deck 的主色必须是稳定的客户/方法论识别色（如 NAVY）——这是红线。

## ④ 对 PPT Engine 落点（能当配色资产 / 还是只参考结构）

**判断：当「数据色资产」可以（小份打包），当「配色规范」不行（结构参考为主）。** 理由：

1. **能直接搬的资产**：130 个彩色色值是**经设计、和谐、跨色相同档亮度对齐**的高质量数据色池——这是我们自己手调要花大力气的部分。可裁剪成一份 `data_palette`（剔掉 cyan、只留 `5~7` 三档主用区）塞进制作层配色资产。MIT 许可无障碍。
2. **不能照搬的部分**：它**缺语义层 + 缺使用纪律 + 缺 PPT 原生格式**。咨询级 deck 真正值钱的「主色唯一 / 强调色仅 3+ 并列才用 / 灰阶 4 级」这套**规范**得来自麦肯锡侧，open-color 补不了。
3. **结构启发（可复用到方法论）**：「色相×亮度档」的二维命名（`oc-blue-6`）是个干净的 token 组织法，我们自建配色规范时可借这个**坐标系结构**（而非借它的色值当门面）。
4. **格式缺口要补**：要进 deck 得自己把选用色值转成 PowerPoint 用法（`.thmx` 主题 / python-pptx 里写死 RGB）。open-color 不给这一步。

**落点一句话**：open-color = **现成的高质量数据色池 + 干净的 token 坐标系结构**，按「剔 cyan、留 5~7 档、只当数据色」裁剪后可入制作层资产；但**主色/语义/纪律仍走麦肯锡规范**，两者拼起来才是咨询级配色。

## ⑤ 坑

1. **「别当主品牌色」是作者亲口红线**（README:14）——色值未来可能改版，绑死品牌识别会被上游变更打脸。我们若打包要**冻结快照版本号（v1.9.1）**，不跟 upstream 漂。
2. **cyan 与明珠规范冲突**：麦肯锡明令废弃 CYAN，open-color 却把 cyan 当正式色相——采用时**必须手动剔除**，否则数据色里混进违规色。
3. **无语义 token**：拿来即用会缺 primary/danger/success 映射，**误以为「装上就有配色系统」是错的**——它只是色库，规范要另写。
4. **感知亮度恒定是「设计声明」非「实测保证」**：README 宣称同档亮度恒定，但**仓库内无对比度/WCAG 数据佐证**（grep `WCAG`/`contrast`/`perceiv` 在 docs 零命中）。当数据色够用，但若 deck 有**无障碍对比度硬要求**，需自己复测前景/背景对比。
5. **无 PowerPoint 原生格式**：给了 Sketch/CSS/LaTeX，**偏偏没 `.thmx`/`.pptx`**——deck 落地要自己搬运色值，不是「导入即用」。
6. **130 色是「池」不是「方案」**：一份咨询级 deck 通常只用 1 主色 + 3~4 强调；直接把 13 色相全摆上＝灾难。它的价值在**挑**，不在**全用**。
