## 一句话：最值得偷的 1 件事

**d3 的 `stack` 生成器把堆叠抽象成「每个点输出一对 `[y0, y1]`（基线/顶线）」，再让你自己拿这对值去画 rect——而瀑布、Marimekko、100% 堆叠柱在 d3 里不是三种图，是 `stack` 换一个 `offset` 函数：`stackOffsetDiverging`（正负分流→瀑布桥图正负段）、`stackOffsetExpand`（归一化到顶线恒=1→Marimekko/100% 堆叠的纵轴算法）、`stackOffsetNone`（零基线→普通堆叠）。** 出处 `docs/d3-shape/stack.md:357/336/378`。这把我们图表手法卡的关键认知点醒了：**waterfall / Mekko / 堆叠柱不该各写一套几何，应共用一个「算 [基线,顶线] 区间」的内核，靠『偏移策略』分叉**——这正是我们 python-pptx 手搓这几张图时该抄的数据模型（d3 的 JS 代码我们用不上，但这个 `[y0,y1]+offset` 的**建模思路**直接可移植）。

## ① 是什么 + 定位 + License

- **是什么**：`docs/what-is-d3.md:42`「**D3 is not a charting library… It has no concept of "charts". When you visualize data with D3, you compose a variety of primitives.**」——它是**底层可视化工具箱**，不是图表库。一个 bar chart 也要几十行代码（文档自嘲引 Amanda Cox：「写一百行代码画个柱状图很正常」`what-is-d3.md`）。
- **架构**：**30+ 个离散 module 的套件**（`what-is-d3.md`「a suite of 30 discrete libraries」）。本地 `package.json` 聚合 **31 个 `d3-*`**：`d3-scale / d3-shape / d3-axis / d3-hierarchy / d3-array / d3-path / d3-selection / d3-format / d3-time / d3-color …`。`d3` 这个包只是「把工具放手边」的便利打包，每个 module 可独立用。
- **关键分层**（对我们最有用）：module 分两类——**碰 DOM 的**（`d3-selection` / `d3-transition`，做 SVG 元素增删改 + data join）和**只算数据不碰 DOM 的**（`d3-scale` / `d3-shape` / `d3-hierarchy`，纯函数：输入数据→输出坐标/路径）。**后者是可跨语言移植的纯算法，正是我们能偷的部分**（`what-is-d3.md`「others (including scales and shapes) only operate on data」）。
- **官方自己劝退**：文档 3 处明示「私有 dashboard / 一次性分析用 d3 是 overkill，去用高层库 Observable Plot」；d3 只为 **NYT / The Pudding 那种「一张图给百万人看、一队编辑打磨」的 bespoke（定制）可视化**值得。→ 对我们是正信号：**高 stakes 咨询级 deck 正属于「bespoke、值得几十行手搓」那一档**，d3 的世界观跟铁律 4 同频。
- **License：ISC**（`LICENSE`，Copyright 2010-2023 Mike Bostock）。ISC ≈ 简化版 MIT，**最宽松一档**：可商用、可改、可闭源，仅需保留版权声明。**对我们抄思路/抄算法零障碍**（对照生态里 AGPL 的项目不能直接抄代码）。⚠️ 但注意：我们偷的是「算法思路」而非 import 它的 JS，license 在我们这几乎不构成约束。

## ② 它做 waterfall / Mekko / Gantt 的思路（核心问题）

**先证实：d3 本体对 `waterfall` / `marimekko` / `mekko` / `gantt` 四个词在 `docs/` + `src/` 中 grep 零命中**——本体**无任何现成 example、无专用 API**，全部 example 在 Observable gallery（外站）上靠原语手搓。这**正面印证资源地图说的「三种都能但要手写」**：d3 给的是积木，不是成品。

下面是「用 d3 原语怎么手搓这三种」的思路（均可移植到我们 python-pptx 手法卡）：

### Waterfall（瀑布/桥图）—— 思路最清晰、最该抄
- **本质**：每根柱不是从 0 起，而是「从上一根的累计值起、画到新累计值」——即**浮动柱**。d3 的 `stack` 天生算的就是这个：`stack.md:75-90` 给出 `stack` 输出 `[[0,3840],[3840,5760],[5760,6720]…]`，每个 `[y0,y1]` 就是一段浮动柱的底和顶。
- **正负分流**靠 `stackOffsetDiverging`（`stack.md:357`「positive values above zero, negative below」）——瀑布里"增项往上叠、减项往下叠"正是这个偏移。
- **渲染**：`stack.md:103-115` 给了**直接拿 `[y0,y1]` 画 rect 的范本代码**：`.attr("y", d => y(d[1])).attr("height", d => y(d[0]) - y(d[1]))`——**「顶线定 y、(底线-顶线)定高」这两行就是浮动柱的全部几何**。→ 我们手法卡里 waterfall 的核心就这两行的 python-pptx 等价物（`rect.top = y(d[1])`、`rect.height = y(d[0])-y(d[1])`）。

### Marimekko / Mekko —— 用 `stackOffsetExpand` + 变宽
- **本质**：纵轴每列归一化到 100%（堆叠占比），横轴每列宽度 ∝ 该列总量。
- **纵轴 100%** 正是 `stackOffsetExpand`（`stack.md:336`「normalizes the values for each point such that the topline is always one」）——一行换 offset 就把堆叠柱变成 100% 堆叠。
- **横轴变宽**：d3 标准做法用 `cumsum`（累计和）手算每列左边界 + 宽度（**band scale 给的是等宽，Mekko 要不等宽，所以这里不能用 band，得手算**——这是 Mekko 比堆叠柱难的唯一增量）。⚠️ 本仓库 docs 无 Mekko 专门页，此横轴算法是 d3 社区通法（gallery 上 `@d3/marimekko` 系列），**非本仓库逐字证据**，标存疑。
- **近亲提示**：`d3-hierarchy` 的 `treemap`（`d3-hierarchy.md`「recursively subdivide rectangles by quantitative value」）也是「按量分割矩形」，跟 Mekko 同族——若做嵌套版 Mekko 可参考 treemap 的递归切分思路。

### Gantt（甘特）—— band scale 定行 + 区间定条
- **本质**：每行一个任务（离散类目），每行画一根「从开始时间到结束时间」的横条。
- **行定位**靠 **band scale**（`docs/d3-scale/band.md:18`「divides the continuous range into uniform bands… typically used for bar charts with ordinal dimension」）：`scaleBand(["任务A","任务B"], [0,高度])` 把离散任务名→均匀行带，`x("任务A")` 返回行起点、`x.bandwidth()` 返回行高。
- **条的水平范围**靠 **time scale**（`d3-scale` 列了 time scale）：`x(开始日期)` 定条左端、`x(结束日期)-x(开始日期)` 定条长。
- → **Gantt = 「band scale 定 y/行高 + time scale 定 x/条长」**，没有专用 API，就是两个 scale 拼。我们手法卡照此建模。

**判定**：跟 PptxGenJS 拆解结论一致——**三种高 stakes 图无现成、必自建**，但 d3 把"自建的数据模型"讲得比谁都清楚（stack 的 `[y0,y1]`、offset 分叉、scale 定位），是**手法卡数据层设计的最佳范本来源**。

## ③ 「数据→视觉编码」范式（scale / axis / shape）对我们图表手法卡的启发

d3 的世界观 = **三段流水线：scale（数据→坐标）→ shape（坐标→几何/路径）→ axis（坐标→可读刻度）**。这套分层值得原样搬进我们的手法卡内核：

### 3A. scale = 「数据域 domain → 视觉域 range」的纯映射（最该抄的抽象）
- `d3-scale.md:1`「**Scales map a dimension of abstract data to a visual representation**」——把"数据值→像素/颜色/尺寸"显式建模成一个对象，而非散落在画图代码里乘乘除除。
- 一张菜单（`d3-scale.md` 列全）：**linear**（数值→长度）、**time**（时间→长度，Gantt 横轴）、**band**（类目→等宽位置，柱/Gantt 纵轴）、**ordinal**（类目→颜色）、**log/pow/symlog**（宽幅数值）、**sequential/diverging**（数值→连续色，热力图/选举图）、**quantize/threshold**（数值→离散档）。
- **启发**：我们手法卡现在大概率把"数据值→EMU 坐标"的换算硬编码在每张图里。d3 的范式提示**抽一个 `scale(domain, range)` 概念**（哪怕只是个小工具函数）：`y = linear_scale([0, 最大值], [图底_EMU, 图顶_EMU])`，让 waterfall/Gantt/柱状**共用同一套定位换算**，而不是各算各的。**band scale 的"等分 + padding"逻辑**（`band.md`：内边距/外边距/对齐）尤其值得抄成柱间距工具。

### 3B. shape = 「坐标→SVG path 字符串」的生成器
- `d3-shape.md:1`「shape generators… each exposes accessors that control how input data are mapped to a visual representation」——line/area/arc/pie/stack 都是"喂数据、吐几何"。
- **关键中转点**：shape 生成器最终产出的是 **SVG path 的 `d` 属性字符串**（`d3-shape.md`：`path.attr("d", line)`），底层是 `d3-path` 模块（`package.json` 含 `d3-path`）。**这串 `d="M…L…C…Z"` 就是④"SVG→DrawingML"路线的中转物**。
- **启发**：pie/arc 的"角度→扇形路径"、area 的"上下边界→闭合路径"，这些**几何计算公式**（圆弧、贝塞尔）跟语言无关，我们做 python-pptx 的自由形状（`freeform`）时可直接抄 d3-shape 的算法。

### 3C. axis = 「scale → 可读刻度」自动派生
- `d3-axis.md:50`：axis 组件吃一个 scale，自动吐刻度线 + 标签的 SVG（`d3.axisBottom(x)`）。`axis.md:60+` 甚至给出**生成的 axis SVG 长啥样**（`<g class="tick">` + `<line>` + `<text>` 的结构）。
- **启发**：轴/刻度**不该手摆**，应从 scale 派生（`scale.ticks()` / `scale.tickFormat()` 自动选"漂亮的刻度数"，`d3-scale.md` 末尾点名这两个 API）。我们手法卡画坐标轴时，"该标几个刻度、刻度值取整到哪"可抄 d3 的 `ticks` 算法思路，避免标出 `0,3333,6666` 这种丑刻度。

**一句汇总**：d3 教我们把图表拆成 **scale（定位）+ shape（成形）+ axis（标尺）** 三层纯函数，**三种高 stakes 图共用前两层、只在数据偏移和形状上分叉**——这比"每张图一个大函数"的写法可复用得多，直接对应我们铁律「复用率 ≥ 50%」的诉求。

## ④ 对 PPT Engine 的落点（含「SVG 渲染再转 DrawingML」可行性）

1. **路线判定：不换栈、只偷思路。** d3 是 JS + SVG/Canvas、运行在浏览器/Node，跟我们 python-pptx（生成 OOXML DrawingML）是**两个世界**。**不要为了图表把制作层切到 JS**（同 PptxGenJS 拆解的结论：切栈丢掉 python-pptx「读改既有 pptx」的独门能力）。**维持 python-pptx 主力的决策再次被佐证。**

2. **「用 d3 渲 SVG 再转 DrawingML」——理论可行但不推荐当主路，列为备选研究项。**
   - 链路：`d3 在 Node 里跑 → 吐 SVG（含 `<rect>`/`<path d="…">`）→ SVG 转 PowerPoint DrawingML`。
   - **断点在第二步**：SVG→DrawingML 没有成熟可靠的开源转换器（这本身可单列一个调研项；LibreOffice headless 能把 SVG 嵌进/转 emf，但保真与可编辑性存疑——**此判断为常识推断，非本仓库证据，标存疑**）。
   - **更现实的折中**：只借 d3 的**纯数据 module（scale/shape/path）算出坐标和 path 字符串**，把这串坐标喂给 python-pptx 画 `shapes.add_shape` / freeform——**即"借 d3 算几何、用 python-pptx 落 XML"**。但这要跨 JS↔Python 进程，工程上不如③「直接把 d3 的建模思路用 Python 重写一遍」干净。→ **结论：优先抄思路（③）用纯 Python 重写；SVG 全链路转换列为低优先备选。**

3. **手法卡数据层直接抄 stack 的 `[y0,y1]+offset` 模型**（②/①一句话）：waterfall/Mekko/堆叠柱共用「算基线-顶线区间」内核，offset 策略分叉。**这是本次拆解给手法卡最硬的可移植产出。**

4. **手法卡内核抄 scale→shape→axis 三层分法**（③）：抽 scale 工具函数让多图共用定位换算；axis 刻度用 `ticks` 思路自动取整。**直接服务"复用率 ≥ 50%"铁律。**

5. **gallery 是手搓范例金矿（但在外站、本地无）**：本地副本只有 docs，**所有 waterfall/Mekko/Gantt 的实际手搓代码都在 observablehq.com/@d3/gallery**（`README` Resources 段指向）。**坑：本地拆不到这些范例**，要看实现得联网去 Observable——这是本次轻拆的覆盖盲区（见⑤）。

## ⑤ 坑 / 注意

1. **本体 ≠ 图表库，没有任何"调一个 API 出一张图"**：误以为 d3 有 `d3.waterfall()` 会扑空（grep 实证零命中）。一切自己 compose。
2. **本地副本是「文档 + 聚合包」，不含范例实现**：`docs/` 是说明文档（很多是 `.vue` 交互演示的源码，掺 Vue/Plot 的样板），**真正的 waterfall/Mekko/Gantt 手搓代码全在外站 Observable gallery**，本地拆不到。要落地实现细节需联网补。
3. **星数/版本对不齐**：资源地图记 113k★；本地副本是 d3 主聚合仓（`package.json` 聚合 31 个子包），**未逐一核对各子包版本**，笔记结论以本地 docs 文本为准。
4. **`stackOffsetExpand` 做 Mekko 只解决了"纵轴 100%"，横轴变宽要另手算 `cumsum`**（band scale 是等宽、给不了不等宽）——别以为换个 offset 就有 Mekko，横轴是额外工作量（②，且此横轴算法非本仓库证据·存疑）。
5. **跨语言移植的是「算法/建模」不是代码**：d3 的 JS 链式 API（`.attr()` / data join）严重依赖 DOM 和 selection，**这套写法在 python-pptx 里无对应、也不该硬搬**；只搬 scale/shape 的"输入数据→输出坐标"纯逻辑。
6. **`d3-axis` 生成的轴是 SVG 元素**，依赖 selection 把它 append 进 `<g>`——轴的"画"那一步绑死 DOM，**只有"该标哪些刻度"（`scale.ticks`）这层纯逻辑可移植**，画的部分得在 python-pptx 重写。
