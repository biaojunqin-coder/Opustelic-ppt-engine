# 开源生态调研 · 两工作流资源地图

> 日期：2026-06-29 · 方法：亲看 PPT master + fan-out 多路 GitHub/联网调研（保真·只收真实项目·star 多为 GitHub API 实测）。
> 目的：为「策略工作流」+「制作工作流」找可复用的素材/思路/底座。服务研究层（生态标准七步法第①步）。
> 一句话结论：**开源界 99% 卡在"制造"（topic→slides），"策略大脑"几乎是空白——这就是我们的护城河。**

---

## 0. 最重要的三个战略判断

1. **没有真正的"策略工作流"**。几十个项目里，只有 2 个碰到方法论边：`PPTAgent`（学术·模仿参考 deck 提 schema）和 `enterprise-ai-skills`（把麦肯锡方法论编码成 prompt skill）。**没有一个做"解读 brief→判目的→市场调研→反复推敲→文字 demo+解读"的完整策略层。** 这正是本项目的差异化定位。
2. **质量基准 = 原生可编辑 PPTX（非图片）**。`ppt-master`(34k) 的核心卖点就是这个；而所有 HTML 底座（Slidev/Marp/reveal.js）导出 pptx **都是图片堆叠、文字不可选**——对我们是硬伤。制作工作流的底座必须走 `python-pptx` 这条"原生"路。
3. **高 stakes 图表（waterfall/Mekko/Gantt）是真门槛**。**没有任何 PPT 库原生支持这三种**；连主流图表库也只有商业版 Highcharts 三种全有。这道坎正是"咨询级"与"通用 PPT"的分水岭——是护城河，也是必须啃的硬骨头。

## 1.5 ⭐ Agentic 流程架构参考（对"两个工作流怎么搭"最有价值）

> 这几个不是看"功能"，是看**流程架构**——它们已经把"PPT agent 该怎么分阶段"试出来了，与 Novel 的"分阶段+出关闸"同构，可直接借鉴。

- ⭐ [sunbigfly/ppt-agent-skills](https://github.com/sunbigfly/ppt-agent-skills) 833★ — **状态机驱动多 agent：Research→Outline→Style→Plan→HTML→Visual QA**·每个子 agent **独立上下文防污染**·**像素级 Visual QA 闭环（自动截图+视觉模型审）**·双 PPTX 导出(PNG 保真+SVG 可编辑)·断点续跑。**几乎是我们"策略→制作+质量门"的现成蓝本。**
  ⚠️ **[2026-07-01 勘误·D6 决策记录]**：此处"蓝本"是首轮资源盘点时凭功能列表下的判断，**未经实际代码采纳验证**。
  同日利用率审计+价值评估显示：它最推荐的 4 个机制（状态机多agent编排/上下文隔离/像素级Visual QA/断点续跑）
  和最推荐的 `svg2pptx.py`，**全部未被采纳进代码**——状态机多agent与 CLAUDE.md 铁律0"一次一个问题·绝不批处理"
  正面冲突、像素级QA是 entry11 已有意识裁剪（Playwright依赖不成比例）、`svg2pptx.py` 被更成熟的 ppt-master
  路线替代。唯一真正吸收的是"分阶段fail-closed"这一大方向哲学的抽象强化，且这更多是与项目自身"以人为主
  全程引导对齐"的最高铁律自然重合，非专门从这个项目移植的具体工程手段。**"蓝本"这个词准确描述的是哲学
  层面的呼应，不是代码层面的移植**——下次评估同类项目时，功能列表吸引人 ≠ 会被真正采纳，需要过一遍
  "跟本项目已拍板的架构铁律兼不兼容"这道筛子，见 `specs/决策记录.md` D6。
- [Westlake-AGI-Lab/Auto-Slides](https://github.com/Westlake-AGI-Lab/Auto-Slides) 520★ ICME2026 — 多 agent 含 **Verification Agent + Repair Agent**（自我纠错=我们 fail-closed 返工的对应物）。
- [KyuDan1/Talk-to-Your-Slides](https://github.com/KyuDan1/Talk-to-Your-Slides) ACL2026 — 5 阶段 Planner→Parser→Processor→Applier→Reporter·**操作结构化对象模型而非像素**(更可控)·带 TSBench 基准(评测思路可借)。
- **启示**：分阶段 + 独立上下文 + 视觉 QA 闭环 + 自我修复，已是 PPT agent 的成熟范式——印证我们工作流设计方向，且 `PPTEval`/`TSBench`/Visual QA 给了"机检质量门"的现成参照。

## 4. 高 stakes 图表方案（护城河关键 · waterfall/Mekko/Gantt）

**核心事实：没有 PPT 库原生支持这三种；得"图表库渲染→图片→python-pptx 插入"或参考原生画法。**

| 图表库 | waterfall | Mekko/Marimekko | Gantt | License |
|---|---|---|---|---|
| [Highcharts](https://github.com/highcharts/highcharts) | ✅原生 | ✅(variwide) | ✅原生 | **商业**(需付费) |
| [D3.js](https://github.com/d3/d3) 113k★ | ✅(手写) | ✅(Bostock 范例) | ✅(手写) | ISC ✅ |
| [Plotly.js](https://github.com/plotly/plotly.js) | ✅原生(最佳) | ❌(open issue) | ~ | MIT ✅ |
| [ECharts](https://github.com/apache/echarts) | ✅(模拟) | ❌ | ~ | Apache2 ✅ |

> 推荐：**免费全能走 D3**（唯一三种都能·要手写）；或 `enterprise-ai-skills`/`Mck-ppt-design-skill` 的 **python-pptx 原生画法**（直接出可编辑图表，最契合"原生"基准）。Highcharts 三种现成但要钱。
> 细化(实测)：waterfall→**Plotly `go.Waterfall` 原生最佳**；gantt→[frappe/gantt](https://github.com/frappe/gantt) 6k★MIT / Plotly `px.timeline`(缺里程碑) / Mermaid 89k★(gantt语法)；mekko→开源极差·只能 Plotly bar 手工模拟。图表→PPTX 桥接有现成轮子 [plotlyPowerpoint](https://github.com/jonboone1/plotlyPowerpoint)。**根因**：python-pptx 不支持这些是因 Office2016 新图表用 `cx:chart` 命名空间、与其架构不兼容(Issue #583 长期未合)——所以"原生可编辑的高级图表"这条最难、最值得啃（= 护城河）。

## 7. 对我们的结论 → 下一步

**怎么用（按工作流落位到八组件×五层）**：
- **策略工作流(对应 /kaishu)**：方法论库 ← `enterprise-ai-skills` 麦肯锡框架 + `issuetrees` 金字塔/SCQA + pitch deck 框架 + 故事框架 gists；调研环节 ← `gpt-researcher`；"outline+用户确认" UX ← 参考 `allweonedev`；质量门 ← 参考 `PPTEval` + tijmenon 的三道审查(action title/水平/垂直逻辑)。
- **制作工作流(对应 /xieshu)**：底座 ← `python-pptx`(原生可编辑·定为铁律)；范本库 ← `Mck-ppt-design-skill` 70 版式；高 stakes 图表 ← D3/python-pptx 原生；素材 ← Tabler+Inter/Noto+咨询配色。
- **质量基准(写进准入)**：①原生可编辑 PPTX 非图片 ②能做 waterfall/Mekko/Gantt ③一页一论点+action title+叙事不断档。

**下一步建议**：① 深读 2-3 个明珠源码（`enterprise-ai-skills` 方法论编码法 + `Mck-ppt-design-skill` 版式实现 + `PPTAgent` 的 PPTEval）→ 提炼进 `specs/PPT方法论/`；② 定"高 stakes deck"的拆解样本来源（拿真实咨询/投资 deck）；③ 把"原生可编辑+三图表"写进 deck_rules 硬门草案。
