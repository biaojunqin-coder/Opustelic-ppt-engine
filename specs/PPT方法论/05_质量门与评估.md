## 一、顶层框架：三维独立评估（PPTEval）

【出处:PPTAgent/README.md L469 abstract】【原文】好 deck = **内容(Content) × 设计(Design) × 连贯(Coherence)**，三者缺一不可、**必须分开度量、不可互相代偿**（内容好不能补设计差）。

| 维度 | 度量粒度 | 五档晋级「关卡」（到 5 分的硬条件）|
|---|---|---|
| **Content 内容** | 逐页评再平均 | 有图(过3)→ 图贴题(过4)→ **图文互补**(到5)。【纯文字 deck 内容分上限 3】 |
| **Design 设计** | 逐页评再平均 | 不冲突可读(1)→ 有彩色(2)→ 有装饰元素(3)→ 和谐少瑕(4)→ **元素增色**(5) |
| **Coherence 连贯** | **整篇**评单分 | 逻辑通(2-3)→ 有基础背景(谁讲/何时/机构)(4)→ **背景齐全+致谢结论**(5) |

【出处:ppteval.py L54-165】【原文/归纳】Overall = 三维均值的均值。**两条可移植的硬约束**：
- **合取式 fail-closed**【ppteval_content.txt L1】【原文】：「每个分档要求该档**所有**标准都达标」——任一标准没到位就掉档。**与本项目「任一项不过就拦」同构。**
- **先描述、后打分（两步解耦）**【ppteval.py L61-80】：先用 VLM 出客观结构化描述，再用 LLM 按 rubric 打分——把「看图理解」和「按标尺评分」拆开，**降主观**。强制输出 `{reason, score}`，不给裸分。

## 二、三道审查门（toolkit · 咨询级 · 可勾选清单）

【出处:gist toolkit 的 2/3/4-review-*.md】【原文】三道独立门，每道 0–100% 评分、**≥90% 过线**、强制输出结构化表 + PASS/REVISE 裁决。

### 门 1 · Action Title 审查【出处:2-review-action-titles.md】
**必过项**：□ 完整句不是标签（"Marketing loses €54K/year" 而非 "Marketing overview"）□ So-what 测试（忙碌高管会在乎吗）□ 主动语态 □ ≤15 词/2 行 □ 有具体数字 □ 不用 m-dash。
**质量项**：□ 无禁用词 □ 不打太极（"could potentially"→"saves"）□ 时态一致 □ 平行结构。

### 门 2 · Storyline 水平逻辑审查【出处:3-review-storyline.md】
**核心测试**：**只读所有标题（不看内容）**，须自成完整叙事，让高管看懂：现状/问题/证据/该做什么/要什么决策。
**带权重维度**：叙事流畅(40%·title 层层递进无逻辑跳步) / SCR 对齐(30%) / 受众视角(20%·倒数第二页前已掌握决策所需) / 结构连贯(10%)。**必产物**：仅靠标题写出的 3–4 句「电梯陈述」。

### 门 3 · Slide Content 垂直逻辑审查【出处:4-review-slide-content.md】
**带权重维度**：垂直逻辑(40%·**每条 bullet 必须「证明 PROVE」其标题**，金句「正文必须让标题成为必然」) / 数据准确(30%·**零编造、每个数字可溯源**) / 完整性(20%) / 质量(10%)。**必产物**：事实核查表（Claim / Source says / Slide says / Match ✅❌）。

## 三、Critic 红黄绿评审闸（麦肯锡）

【出处:ai-skills/mckinsey-critic】【原文】三级评级：**Green**=可交付仅小修 / **Yellow**=连夜可修别照发（有损可信度的结构问题，**最常见**）/ **Red**=核心论点或结构坏了需返工。

四维校验：**Structure**（叙事是否 Problem→Context→Analysis→Solution）/ **Rigor**（每条论断有数据来源、数字具体、比较公平）/ **So-What**（每节有 takeaway、只读标题能懂全论点、推荐可执行且有时限）/ **Completeness**（给选项有对比框、竞品指名、下一步有 owner+时间）。

【原文，两条用法纪律】「**先跑 critic 再美化排版**——先修结构再变漂亮」；critic 是**对抗性的、只给 Top 3 fixes + 指出一个有效处以便保护**。→ **与本项目「防假绿」天然同构。**

## 四、★机检 vs 人审分工（铁律2 的直接落料）

把上面三套门的条目，按「能不能程序确定性判定」拆两类：

### 【机检】可做 fail-closed 脚本硬门（不过就拦，不靠 LLM 打分）
- **禁用词五连**：leverage / harness / transformative / synergy / paradigm —— 命中即拦。【toolkit，跨门反复】
- **禁 m-dash（—）**：命中即拦。【toolkit】
- **字数/段数上限**：标题 ≤15 词；版式 char_budget（见 [02_页型手法](02_页型手法.md)）。【toolkit + Mck layout-matrix】**⚠️2026-07-01 yanjiu核实：`check_page_budget`/`check_font` 代码已写(`rules.py` L65/74)+单测覆盖，但从未接入 `engine/pipeline.py::build_deck` 生产调用链(只有 `run_text_gate` 被接线)——"已落地成硬门"的表述目前对生产路径不成立，见§七①。**
- **数字必带来源标注**：每个数据声明须有 source tag，无则拦（零编造的可机检代理）。【toolkit 门3 + ai-skills/synthesis】
- **constraint verify（纯规则）**：页数区间、宽高比(16:9/4:3)、语言是否符合要求，返回 0–1 命中率。【出处:typings.py L99-116，2026-07-01 联网核实：**非** EMNLP2025 论文原始方法——该文件 GitHub 仅一条 commit(2026-05-19 "feat: add eval-diversity+constraint")，比论文晚一年半，与同组作者后续论文 DeepPresenter(arXiv:2602.22839) 逐字对应，论文全文未提及。此前【PPTEval】标法属过度归因，见§七】
- **结构完整性**：每页有 source / 有 speaker notes / 相邻页不同版式 / 章节页标题与目录逐字匹配。【Mck planning-guide + PPTAgent category_split】**⚠️2026-07-01 yanjiu核实："相邻页不同版式"这一条全仓搜索无任何实现(非走样，是从未开工)，见§七🟢采纳。**
- **合取式判档**：任一必过项不达标 → 掉档/拦截。【PPTEval】

### 【人审 / 盲审 agent】需审美与判断（必引证据，绝不 LLM 裸分当硬信号）
- **So-what 测试**：忙碌高管会在乎吗（含义/影响）。
- **Claim-title 质量**：标题是「带数字可证伪的论断」还是「主题标签」。
- **水平逻辑**：只读标题能否复述出推荐 + 走 S/C/R。
- **垂直逻辑**：每条 bullet 是否真「证明」标题。
- **Critic 红黄绿**：结构/严谨/so-what/完整性四维综合裁决。

> **本项目纪律加严**：明珠允许「自动判定 passed:true 即过」；**本项目更严——自省最高只给 🟡，🟢 必须人工自测**（防假绿，比明珠多一道）。

## 五-附、开源应用质量门交叉印证（拆 40 项目实证 · 反复坐实防假绿是护城河）

> 拆 40 个开源项目的最强元结论：**质量门是行业最普遍的短板**。料在 [开源拆解](../../research_lib/开源拆解/README.md)。

**反面 · fail-open（生成即标 success → 正是我们差异化空间）**：
- `Auto-Slides`：Verification Agent **写了却没接进主流程**，主流程验证不过只弹 input 问用户（=没闸）。
- `presentation-ai`/`LandPPT`/`pptmaker`/`banana-slides`/`slide-deck-ai`：**人眼当唯一闸 / 无内容质量校验**，单页失败照样标 completed。
- 调研类 `gpt-researcher`/`Perplexica`：fact-check 是 **LLM 自评**（照搬当 🟢 正踩防假绿）。
→ 几乎所有应用都缺"内容/数字质量硬闸"。**我们「🟢必须人工自测 + fail-closed 机检」是护城河。**

**正面 · fail-closed 范式（可抄进 `reinforce/deck_rules`）**：
- `guizang`：110 行**静态正则校验器** `exit(1)` 真 fail、零 LLM 打分，拦 9 类 → 最纯机检硬门。
- `Instrumenta` **Slide Grader**：38 项分级检查（6 类×三级严重度）+ **自动修严格白名单**（只 9 项敢自动改、设计判断只报警不替人拍板）→ PPT Linter 范本。
- `ppt-master`：黑名单 block + **故意无 auto-fix 强制 LLM 带上下文重做**。
- `Auto-Slides 编译层`：retry≤5 + `returncode==0 且 PDF 真实存在`才过 → 有**非 LLM 客观二值判据**的闸才成立。
→ 共性：**机检只处理机械确定项（exit 码/正则/客观二值），设计判断留人**——铁律2「机检/人审分工」的实证模板。
